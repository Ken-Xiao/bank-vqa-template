# 董办银行分析工具产品设计路线图

## 一、产品定位

目标不是做一个静态财报看板，而是做一个面向银行董办、战略部、财务部和资本市场沟通团队的“对标分析 + 经营解释 + 汇报生成”工具。

核心交付包括：

1. 选择目标银行、对标银行、银行类型和年份后，自动生成经营诊断。
2. 每张图都回答一个经营问题，而不是只呈现指标。
3. 报告语言围绕 VQA 框架展开：价值质量、经营机制、资本市场验证、行动优先级。
4. 支持导出董办汇报版报告和多工作表数据底稿。

## 二、推荐产品结构

| 层级 | 模块 | 说明 |
| --- | --- | --- |
| 数据层 | 银行主数据 | 银行简称、类型、区域、年份、样本资格、别名映射 |
| 数据层 | 财务指标库 | 盈利、息差、资产负债、风险、资本、流动性、估值 |
| 数据层 | 派生指标库 | 息差对冲缺口、真实非息、现金利润、RWA增速缺口、零售风险分化 |
| 分析层 | VQA诊断引擎 | 价值质量评分、质量折价/价值错配判断、预警等级 |
| 分析层 | 专题分析引擎 | 盈利真实性、负债结构、零售风险、对公行业风险、投资资产、资本消耗 |
| 展示层 | 董办分析页面 | 选择口径、图表导航、数据覆盖、章节故事线、图表解释 |
| 输出层 | 演示报告 | 封面、故事线、章节承接页、图表页、行动建议页 |
| 输出层 | 数据底稿 | 选择摘要、目标银行、对标银行、类型均值、完整性、全样本明细 |

## 三、页面体验建议

### 1. 选择阶段

选择前应显示完整银行池，突出“目标银行、对标银行、类型参照、分析年份”四个动作。确认后隐藏大选择池，只保留已选结果框和重新开始按钮，避免页面被筛选器占满。

### 2. 分析阶段

页面按汇报逻辑组织，而不是按字段堆叠：

1. 方法论：说明 VQA 如何同时看经营端和资本市场端。
2. 数据覆盖：先披露指标完整性，说明哪些结论可以进入主报告。
3. 价值质量总览：给出 30 秒结论。
4. 盈利质量：回答利润是不是主业修复。
5. 息差负债：回答负债端能否对冲资产让价。
6. 风险拨备：回答风险是否充分确认。
7. 资本估值：回答扩表是否创造价值，PB 是价值错配还是质量折价。
8. 行动建议：按“现在、近期、中期”给出动作顺序。

### 3. 报告阶段

导出报告应模拟 PPT 叙事：

1. 封面
2. VQA 方法论
3. 本次选择口径
4. 总故事线
5. 章节承接页
6. 一图一页
7. 图下故事说明
8. 行动建议
9. 数据覆盖附录

## 五、AI能力接入建议

当前页面已经具备“数据驱动话术”的基础，但真正的 AI 版本建议采用两段式：

1. 先由规则和计算引擎生成结构化事实：目标值、对标均值、类型均值、分位、同比、缺口、预警标签。
2. 再把结构化事实交给 AI 生成自然语言：核心结论、经营机制解释、对标差异、管理动作、汇报口径。

这样做可以避免 AI 直接编数，也能让董办报告保留稳定口径。

## 八、下一轮迭代设计：产品、展示、分析内容一起推进

### 1. 产品端：从“生成报告”升级为“董办对标工作台”

下一轮产品目标不是只多放图，而是让董办可以围绕一次汇报任务完成选择、诊断、复核、成稿和留痕。

| 模块 | 需要新增的能力 | 价值 |
| --- | --- | --- |
| 分析项目 | 保存目标银行、对标银行、银行类型、年份和报告版本 | 同一家银行可以复用历史对标口径 |
| 对标组模板 | 内置“同区域、同类型、同规模、资本市场关注度”四类对标组 | 降低每次手工选银行的成本 |
| 诊断状态 | 显示 VQA 总分、薄弱维度、数据覆盖、报告完整度 | 让董办先判断报告是否足够进入汇报 |
| 图表取舍 | 每张图可加入主报告、进入附录、隐藏 | 解决不同汇报对象关注点不同的问题 |
| 口径复核 | 每个派生指标显示来源字段、计算公式、缺失情况 | 让财务、战略、董办可以共同校验口径 |
| 报告版本 | 保存“经营汇报版、资本市场沟通版、董事会精简版” | 同一套数据生成不同叙事强度的报告 |

### 2. 展示端：从“网页截图感”升级为“PPT 汇报母版”

展示端下一轮要重点处理三件事：选择页识别度、网页叙事连贯性、导出 PDF/PPT 的正式感。

| 场景 | 下一步改法 | 设计要求 |
| --- | --- | --- |
| 选择页 | 做成左右分区：左侧选择目标银行，右侧维护对标银行和类型均值，下方固定显示已选结果 | 已选银行必须像标签/名片一样清晰可见 |
| 网页主线 | 在每章前增加“本章回答什么问题、为什么重要、看哪几张图” | 页面要能把故事说清楚，而不是只堆图 |
| 图表页 | 每张图固定为“主标题、辅标题、图、关键结论、机制解释、行动启示” | 图下文字要解释原因和管理含义 |
| 导出报告 | 继续向苏农汇报材料靠齐，形成封面、故事线页、章节页、一图一页、结论页、附录页 | PDF 每页必须在一页内完整显示 |
| PPT 输出 | 从浏览器打印 PDF 升级为真正 `.pptx` | 方便董办继续编辑、删页、替换话术 |

### 3. 分析内容端：从“规则话术”升级为“结构化事实 + AI 解读”

分析内容下一轮要把现有数据继续转成更有信息量的经营判断，再让 AI 基于结构化事实生成讲解。

| 分析主题 | 可继续生成的指标 | 应回答的问题 |
| --- | --- | --- |
| 息差对冲 | 生息资产收益率变化、计息负债成本率变化、息差对冲缺口、真实存贷利差 | 资产端让价是否被负债端成本改善对冲 |
| 盈利真实性 | 核心营收增速、真实核心非息、手续费资产比、高波动收入占比、现金利润比 | 利润是主业改善、拨备释放，还是波动收入贡献 |
| 风险确认 | 隐性不良、逾期偏离度、零售分产品不良、拨备覆盖、利润拨备缺口 | 风险有没有充分暴露，后续利润是否仍有压力 |
| 资本效率 | RWA 增速、资产增速、利润增速、CET1 缓冲、资本充足率缓冲 | 扩表是否消耗资本但没有形成回报 |
| 估值验证 | PB、ROA、核心营收、风险指标、资本缓冲 | 低估值是价值错配还是质量折价 |
| 董办行动 | 当前优先级、短期修复动作、中期能力建设、资本市场沟通口径 | 结论如何转成董事会和投资者都能理解的动作 |

建议下一轮把 AI 生成拆成两步：第一步由前端/数据层生成事实包，包含目标值、对标均值、类型均值、分位、同比、缺口和预警；第二步由 AI 只读取事实包生成解释，避免 AI 自己编数字。最终每张图都应该对应一段“为什么这样、说明什么、下一步做什么”的董办话术。

## 九、三条线的落地顺序

### 1. 产品端

先把工具从“单次生成”变成“项目工作台”。第一阶段保存分析项目和对标口径；第二阶段增加默认对标组和报告版本；第三阶段补充口径复核、图表取舍、报告留痕。这样董办每次汇报都可以回到同一个项目继续迭代，而不是重新选择一遍。

### 2. 展示端

先把网页端做成可阅读的汇报故事，再把导出端升级成可编辑演示文稿。第一阶段优化左侧图表导航和章节故事线；第二阶段固定每张图的“主标题、辅标题、图、关键结论、机制解释、行动启示”，并先导出独立 HTML 演示报告作为中间稿；第三阶段基于 HTML 中间稿转换 PDF 和真正 `.pptx`，其中 PDF 作为定稿版本，PPTX 作为可编辑汇报版本。

### 3. 分析内容端

先完善事实包，再接入 AI 写稿。第一阶段为每张图生成目标值、对标均值、类型均值、同比、分位和缺口；第二阶段把指标组合映射为经营机制解释；第三阶段让 AI 基于事实包生成不同版本话术，包括董事会汇报版、资本市场沟通版和管理层行动版。

## 十、后续三条线迭代路线

### 1. 产品端路线：从单页工具到董办工作流

| 阶段 | 时间优先级 | 核心目标 | 具体交付 |
| --- | --- | --- | --- |
| P0 | 近期 | 让一次分析可以完整闭环 | 分析项目保存、已选口径复用、HTML演示报告导出、选定数据底稿导出 |
| P1 | 中期 | 让董办可以管理多套对标口径 | 默认对标组、银行画像页、报告版本管理、图表加入主报告/附录/隐藏 |
| P2 | 后期 | 让工具进入正式汇报流程 | 多人协作复核、指标口径审批、报告留痕、历史版本对比 |

产品端最先要做的是“项目化”。董办不会只生成一次报告，而是会反复调整对标银行、删改图表、复核口径、更新年份。因此下一步应把当前选择状态沉淀为一个“分析项目”，每个项目保存目标银行、对标银行、银行类型、年份、报告版本、选中图表和导出记录。

### 2. 展示端路线：从页面看板到汇报母版

| 阶段 | 时间优先级 | 核心目标 | 具体交付 |
| --- | --- | --- | --- |
| P0 | 近期 | 让网页端故事线更顺 | 章节前言、图表导航只保留图名、每图一页式报告预览、HTML演示报告 |
| P1 | 中期 | 让导出物接近正式PPT | PPT式封面、章节承接页、图表页、结论页、数据附录页统一母版 |
| P2 | 后期 | 让输出物可编辑可复用 | HTML转PDF、HTML转PPTX、PPT母版配置、不同报告版本一键切换 |

展示端下一步的重点不是继续堆组件，而是统一“汇报语法”。每个章节都要先回答“本章为什么重要”，每张图都要固定成“主标题、辅标题、图表、关键结论、机制解释、行动启示”。导出的 HTML 是中间稿，PDF 是定稿，PPTX 是可编辑汇报稿。

### 3. 分析内容端路线：从规则解释到事实驱动 AI

| 阶段 | 时间优先级 | 核心目标 | 具体交付 |
| --- | --- | --- | --- |
| P0 | 近期 | 先把事实包做扎实 | 每张图生成目标值、对标均值、类型均值、同比、分位、缺口、数据完整性 |
| P1 | 中期 | 把事实转成经营机制解释 | 盈利真实性、息差对冲、风险确认、资本效率、估值验证五类专题解释器 |
| P2 | 后期 | 让 AI 生成可控报告语言 | 董事会版、资本市场版、管理层行动版三套话术；事实引用和数字溯源 |

分析内容端要避免让 AI 直接“看图编话”。正确路径是先由数据层生成事实包，再让 AI 只基于事实包写解释。每张图的事实包至少包括：目标银行当前值、对标银行均值、类型均值、过去一年变化、五年变化、全样本分位、缺口方向、数据完整性、推荐解释角度。这样生成的话术既能更像顾问报告，也能回到底层数据复核。

### 4. 推荐开发顺序

1. 先做产品端 P0：分析项目保存、选中图表状态、HTML演示报告中间稿。
2. 同步做展示端 P0：网页章节前言、每图一页预览、报告页统一母版。
3. 再做分析内容端 P0：为每张图生成结构化事实包。
4. 然后进入 P1：默认对标组、正式PPT母版、五类专题解释器。
5. 最后进入 P2：多人复核、HTML转PPTX、AI多版本话术。

短期最关键的里程碑是：用户选择一家目标银行和一组对标银行后，系统能稳定生成一份“网页可读、HTML可下载、PDF可定稿、PPTX可编辑”的完整汇报，并且每个结论都能追溯到事实包和底层指标。

## 十三、当前体验优化迭代

| 模块 | 优化动作 | 目的 |
| --- | --- | --- |
| 选择区 | 目标银行、对标银行、年份、类型、报告版本、对标模板改为两行栅格 | 解决初始选择区横向挤压和出边框问题 |
| 图表导航 | 左侧导航仅保留“查看更多”中的图片图表名称 | 避免导航混入章节、主图和解释性文字 |
| HTML演示报告 | 新增经营诊断摘要页和五类专题解释页 | 报告不再只是单句故事页，而是包含数字、评论和专题判断 |
| 下一步 | 基于 HTML 中间稿继续做 PDF 定稿和 PPTX 可编辑稿 | 保持“网页报告 → HTML → PDF/PPTX”的输出链路 |

## 十四、客户版语言与数据钻取迭代

| 模块 | 迭代动作 | 面向客户的变化 |
| --- | --- | --- |
| 汇报语言 | 将“模板、AI评价、建议生成”等内部表达替换为“董事会汇报、董办解读、管理建议、关键事实” | 页面文字可直接用于客户董事会沟通 |
| 数据呈现 | 数据覆盖区升级为“数据类型 → 字段 → 年度变化”三段式钻取 | 董办可先看数据主题，再下钻字段，并查看目标银行与对标银行的时间变化 |
| 报告母版 | HTML/PDF 演示报告新增关键数字页、董办评论框、五类专题解释页和更强的蓝色汇报母版 | 输出报告更接近正式咨询汇报材料，而非网页截图 |
| 专题深化 | 后续围绕盈利真实性、息差对冲、风险确认、资本效率、估值验证继续扩展专题页 | 每个专题都形成“关键事实、机制解释、管理动作”的固定结构 |

## 十五、数据钻取与默认对标组迭代

| 模块 | 迭代动作 | 当前规则 |
| --- | --- | --- |

| 专题深化 | 页面新增五类专题路线 | 盈利真实性、息差对冲、风险确认、资本效率、估值验证继续作为后续专题开发主线 |

## 十六、后续迭代分析：产品、展示、分析段

### 1. 产品端

下一步应从“单次分析工具”升级为“董办项目工作台”。优先级建议如下：

| 优先级 | 迭代方向 | 具体能力 |
| --- | --- | --- |
| P1 | 项目版本管理 | 保存多个分析项目、报告版本、对标组、入选图表和导出记录 |
| P1 | 对标组治理 | 支持同区域、同类型、同规模、资本市场相近四类默认对标组，并允许手工微调后保存 |
| P2 | 口径复核 | 每个指标展示来源字段、计算公式、缺失原因和可用年份 |
| P2 | 协作流程 | 董办、财务、战略分别对数据口径、结论语言和行动建议进行复核 |
| P3 | 后台配置 | 配置银行主数据、区域划分、默认对标规则、报告版本和禁用词 |

### 2. 展示端

展示端要继续向正式董事会材料靠拢，不只是网页可看，而是能直接成为汇报母版。

| 优先级 | 迭代方向 | 具体能力 |
| --- | --- | --- |
| P1 | HTML报告母版 | 增加封面、目录、章节摘要、关键数字页、专题解释页、图表页、行动页、附录页 |
| P1 | 数据钻取体验 | 数据类型、字段列表、年度变化分步进入，字段全部中文展示 |
| P2 | PPTX输出 | 基于 HTML 中间稿生成可编辑 PPTX，并保留 RSM/苏农汇报风格母版 |
| P2 | 图表排版 | 每张图支持“图 + 右侧评论框”或“上方数字条 + 图 + 下方建议”两类版式 |
| P3 | 董事会模式 | 一键切换为无编辑控件、适合投屏和汇报的阅读模式 |

### 3. 分析内容端

分析段应从“五类解释器”继续向“专题结论引擎”升级。

| 优先级 | 迭代方向 | 具体能力 |
| --- | --- | --- |
| P1 | 专题事实包 | 每个专题固定输出目标值、对标均值、类型均值、分位、同比、五年变化、完整性 |
| P1 | 专题页深化 | 盈利真实性、息差对冲、风险确认、资本效率、估值验证分别形成独立专题页 |
| P2 | AI写稿 | AI只读取事实包，生成董事会版、资本市场版、管理层行动版三套文字 |
| P2 | 结论校验 | 每条结论绑定引用指标，支持点击回到底层字段和年度趋势 |
| P3 | 策略建议库 | 将行动建议拆成资产负债、客户经营、风险分类、资本规划、市场沟通五类管理动作 |

短期最应该继续做的是：先把“专题事实包 + 专题页深化”做扎实，再接 PPTX 输出。否则 PPTX 只是更漂亮的页面导出，分析深度不会明显提升。

## 十七、分析内容端升级：专题主体与智能解读

本轮先把分析内容端从“图表解释”推进为“专题主体”。每个专题不再只是一段说明，而是固定包含董事会问题、关键指标事实包、机制解释、董办解读和管理动作。

| 模块 | 已落地动作 | 后续升级方向 |
| --- | --- | --- |
| 专题主体 | 新增盈利真实性、息差对冲、风险确认、资本效率、估值验证五类可切换专题 | 后续可继续增加区域竞争、负债客户、零售风险、投资收益波动等专题 |
| 事实包 | 每个专题输出目标值、对标均值、类型均值、一年变化、五年变化、分位和完整性 | 后续为每个事实增加来源字段、计算公式和点击追溯 |
| 智能解读 | 按董事会版、资本市场版、管理层行动版生成三套客户可读文字 | 后续接入大模型时，只把事实包和专题问题作为输入，避免无依据写稿 |
| 数据导出 | 数据工作簿新增“专题AI解读”sheet，沉淀每个专题的事实和文字 | 后续支持客户复核后回写批注，形成正式报告底稿 |

下一步分析内容端建议优先做两件事：第一，把每个专题的指标权重和红黄绿阈值配置化；第二，让每条AI解读都能点击回到具体指标、图表和年份趋势。

## 十八、三个 P0 迭代落地

本轮先按产品、展示、分析内容三个方向各落一个 P0，目标是让工具更接近董办真实工作流。

| 方向 | P0能力 | 已落地内容 |
| --- | --- | --- |
| 产品端 | 分析项目摘要 + 报告生成流程 | 页面新增当前项目摘要，显示目标银行、对标银行、类型均值、报告版本、当前专题、纳入图表数和 VQA 诊断；同时新增“选择口径、生成诊断、选择专题与图表、导出报告与底稿”四步流程 |
| 展示端 | 专题主体汇报版式 | 专题页从单列表升级为“专题判断、证据指标、一句话结论、事实表、三类解读”的正式汇报结构 |
| 分析端 | 专题红黄绿判断 + 证据引用 | 每个专题根据指标分位生成红黄绿判断，并把关键证据指标写入董事会版、资本市场版、管理层行动版解读 |

下一轮建议把本轮的专题判断进一步配置化：每个专题的红黄绿阈值、核心指标权重、必须引用指标和禁用表述都应进入配置文件，便于后续接入真正的大模型写稿。
