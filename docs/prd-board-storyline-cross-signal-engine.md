# PRD：董事会经营质量故事线与交叉信号引擎

版本：v11.0  
日期：2026-06-06  
适用项目：BenchmarkIQ 银行经营对标分析平台  
来源：结合产品全景诊断文本与“经营质量诊断故事线”PRD 草案整理。

## 一、产品定位更新

BenchmarkIQ 下一阶段不再定位为“银行指标对标工具”，而应升级为“银行经营质量诊断工作台”。核心价值不是把指标排出来，而是在银行董办、管理层或研究员收到年报数据后的 48 小时内，完成从数据到判断、从判断到证据、从证据到行动的压缩。

目标用户先按 To B 场景收敛：

1. 银行董事会/董办：8 分钟内完成设定口径、看总判断、确认证据、导出上会材料。
2. 银行管理层/CFO/CRO：看到差距形成机制、行动优先级和 0-12 个月复核路径。
3. 咨询/研究分析师：进入专题深钻、数据复核、口径差异和报告编辑。

To C 投资分析入口暂不作为当前主线，但保留资本市场版语言和 PB/估值深钻能力。产品主入口应优先服务银行决策工作台，而不是同时承载行业筛选器。

## 二、核心问题

当前系统已经具备 SPARC/VQA、对标组、三源数据、专题深钻、AI/本地模板写稿和报告导出，但仍存在四个断点：

1. 功能很多，但用户进入后仍需要理解工具结构，而不是沿着“设定口径 -> 看结论 -> 出报告”的主线前进。
2. 文字解读仍有时偏方法论说明，缺少金融分析中“一眼可得”的结论句。
3. 证据地图、专题、报告层虽已同源化，但跨维度因果推理还不够强，容易形成平铺指标而非诊断链。
4. 三源数据已经形成 ready 层和抓取明细库，但新增数据价值还没有充分转化为董事会故事线中的“为什么”和“怎么办”。

因此本 PRD 的目标是建立：

**数据 -> 位置 -> 方向 -> 原因 -> 行动 -> 验证** 的完整链条。

## 三、产品主线：三幕九章

董事会场景采用 SCR 结构：Situation 定位、Complication 归因、Resolution 行动。在线 Portal、正式 HTML/PDF、PPTX 应共享同一章节模型。

| 幕 | 章 | 页面问题 | 核心输出 | 优先级 |
|---|---|---|---|---|
| Situation | S1 | 经营质量总览：一句话能说清吗？ | 经营质量总分、五维状态、一句话判断、最紧迫矛盾 | P0 |
| Situation | S2 | 在同行中处于什么位置？ | ROA × 经济利润/分位散点，目标银行、对标组、类型均值 | P1 |
| Situation | S3 | 哪些信号最值得董事会关注？ | Top 3 Changes & Deviations 卡片，当前值、同比、vs peer、含义 | P0/P1 |
| Complication | C1 | 盈利是真赚还是虚胖？ | 营收分解、盈利真实性三角、ROA 归因 | P1 |
| Complication | C2 | 息差还能撑多久？ | NIM 桥接瀑布、资产端/负债端/结构效应、定期存款趋势 | P1 |
| Complication | C3 | 风险是清了还是藏了？ | NPL × 逾期偏离度四象限、隐性不良、拨备余量 | P1 |
| Complication | C4 | 资本效率在为谁服务？ | ROE/ROA/资本成本/经济利润、市净率理论值 vs 市场值 | P1 |
| Resolution | R1 | 管理层的优先级应该怎么排？ | 行动优先级矩阵，影响力 × 紧迫性，责任方向和时间窗口 | P1 |
| Resolution | R2 | 如果改善了，价值会怎么变？ | What-if 情景，NIM/NPL/中收变化对 EP 和理论 PB 的影响 | P1/P2 |

## 四、关键新增能力：交叉信号引擎

### 4.1 定义

交叉信号引擎用于把单指标对标升级为跨维度推理。每个核心判断不仅看本维度指标，还要引用其他维度作为证据或反证。

示例：如果 ROA 偏弱，系统不只说盈利低，而要判断是收入端不足、风险消耗、成本效率，还是资本占用导致。若 NIM 下行同时定期存款占比上升，解读应直接指向“负债成本刚性”，而不是泛泛描述息差压力。

### 4.2 需要新增的因果链

建议新增 `config/causal_chains.json`，定义以下链路：

| 因果链 | 触发条件 | 要回答的问题 | 关键指标 | 输出 |
|---|---|---|---|---|
| `nim_decomposition` | NIM 低于对标或同比下降 | 息差收窄来自资产端还是负债端？ | 生息资产收益率、计息负债成本率、定期存款占比、活期存款占比 | NIM 桥接数据和主因判断 |
| `roa_attribution` | ROA 低于对标 | ROA 偏低是收入不够还是风险吃掉利润？ | 核心营收增速、手续费资产比、不良率、拨备覆盖率、成本收入比 | ROA 归因排序 |
| `hidden_risk_exposure` | 逾期偏离度 > 1 或隐性不良高于对标 | 表面不良率可信吗？ | 不良率、逾期偏离度、隐性不良暴露率、关注类贷款、拨备覆盖率 | 风险确认结论 |
| `value_creation_check` | 默认执行 | 这家银行在创造价值还是消耗价值？ | 经济利润、ROE、资本成本、市场 PB、理论 PB、PB 偏离 | 价值创造判断 |

### 4.3 计算函数需求

新增或扩展以下函数，尽量采用增量方式，不破坏现有 `fallbackVqaEngine`、`layeredTopicFactModel` 和 `reportDeliveryModel`。

| 函数 | 位置建议 | 说明 |
|---|---|---|
| `runCausalChain(chainId, row, peers, allRecords)` | `js/05-analysis.js` 或新模块 | 执行单条因果链，返回触发状态、证据指标、主因、置信度、叙事输入。 |
| `runAllCausalChains(row, peers, allRecords)` | `js/05-analysis.js` 或新模块 | 批量执行并按偏离强度排序。 |
| `buildCrossSignalMatrix(row, peers)` | `js/36-layered-fact-model.js` 或新模块 | 输出五维之间的联动异常矩阵。 |
| `topNChangesAndDeviations(row, peers, allRecords, n)` | `js/05-analysis.js` 或新模块 | 按 `abs(z_vs_peers) * 0.6 + abs(z_vs_self_3yr) * 0.4` 排序。 |
| `buildNimBridgeData(row, prevYearRow)` | `js/03-data-format.js` 或新模块 | 生成 NIM 瀑布图数据；缺去年数据时自动降级为水平对标。 |

## 五、数据层更新

### 5.1 七层数据管道

| 层级 | 角色 | 当前基础 | 本 PRD 更新 |
|---|---|---|---|
| Layer 0 原始数据源 | 原主数据、Tushare、年报抓取 | 已有 | 保持三源职责分工，不互相硬替换。 |
| Layer 1 字段标准化 | bank/year/标准字段名 | `build_ready_data_layer.py` | 增加因果链所需字段的治理状态。 |
| Layer 2 指标计算 | 派生指标 | `js/03-data-format.js`、ready 层 | 补 EP、NIM 桥接、隐性风险、PB 偏离、重定价缺口。 |
| Layer 3 对标引擎 | 分位、均值、类型参照 | `js/05-analysis.js` | 保持现有逻辑，作为证据输入。 |
| Layer 4 因果拆解 | 跨维度推理 | 部分已有机制归因 | 新增 Cross-Signal Engine。 |
| Layer 5 叙事生成 | 本地模板/LLM | `js/12-ai-narrative.js`、`js/33-llm-commentary.js` | 每章按 SCR 结构生成观点、证据、机制、管理含义。 |
| Layer 6 呈现渲染 | Portal/HTML/PPTX/PDF | `reportDeliveryModel` 已有 | 九章故事线共享章节模型。 |
| Layer 7 交付验证 | 页序、引用、语言纪律 | 现有契约测试 | 增加章节标题、数字数量、引用字段和 NIM 闭合校验。 |

### 5.2 三源数据职责

| 数据源 | 职责 | 不能做什么 |
|---|---|---|
| 原有主数据 | 默认诊断、趋势、对标均值、核心报告主线 | 不被年报抓取直接覆盖。 |
| Tushare | 市场估值、标准三表、股息率、市值、换手率、部分成熟比率 | 不混入年报披露口径，不替代监管附注。 |
| 年报抓取 | 2025 年报核验、监管指标、零售分产品、期限桶、IFRS9、核销回收、附注明细 | 不直接硬塞进前端宽表，先聚合为专题视图和明细 drill 表。 |

## 六、页面与报告更新

### 6.1 Portal

1. Launch 只承担设定口径，不展示过多方法论。
2. Executive 默认显示 S1 + 三个董事会问题 + 本地证据解读。
3. Evidence Map 升级为 S2/S3：同业位置、Top Changes & Deviations、估值锚和因果链入口。
4. Topics 不再只是专题卡片，应承接 C1-C4 四类归因专题。
5. Report Studio 读取同一 `reportDeliveryModel`，显示九章故事线的章节状态、引用状态和可导出状态。
6. Data & Validation 显示三源字段治理、年报核验、因果链字段缺口和可替换策略。

### 6.2 报告

正式报告页序建议：

1. 封面
2. 执行摘要：S1 总览
3. 行业位置：S2 同行位置
4. 关键变化：S3 Top 3 信号
5. 盈利真实性：C1
6. 息差防守：C2
7. 风险确认：C3
8. 资本效率与估值：C4
9. 行动优先级：R1
10. 情景推演：R2
11. 数据口径与附录

PPTX 和 HTML/PDF 必须使用同一章节数组。PPTX 可先做可交付版，不要求高保真矢量化。

## 七、语言与生成规则

### 7.1 三种语言通道

| 通道 | 结构 | 风格 |
|---|---|---|
| 董事会版 | 问题 -> 证据 -> 方向 | 少数字、强观点、可讨论。 |
| 资本市场版 | 判断 -> 证据 -> 估值含义 | 关注 PB、ROE、EP、分红、市值和投资者叙事。 |
| 管理层行动版 | 发现 -> 差距 -> 行动 -> 预期影响 | 明确时间窗口、责任方向和复核指标。 |

### 7.2 语言纪律

1. 标题必须是问句或判断句，禁止“盈利分析”“风险概览”等描述性标题。
2. 每段最多 5 个数字，超过部分进入脚注或数据表。
3. 每段强结论至少绑定 2 个 factPack 指标。
4. 如果数据来源为 `missing / partial / validation-only`，语言强度必须降级。
5. 不解释系统方法论，不写“需要综合分析”等空话；应直接基于当前银行和对标结果给结论。

| Sprint | 交付内容 | 验收 |

## 九、验收标准

以目标银行“宁波银行”、对标组“南京银行、杭州银行、上海银行、江苏银行”、年度 2024 为验收样例：

1. 60 秒内看到 S1 总览页，一句话判断包含银行名称、年度和最紧迫矛盾。
2. S3 展示 3 张信号卡，每张包含当前值、vs peer 偏离和一句管理含义。
3. C2 NIM 桥接图中资产端、负债端、结构效应之和能闭合到 NIM 同比变化；若缺去年数据，页面清楚标注降级。
4. C3 风险四象限点位与手工计算一致。
5. R1 第一优先级行动来自偏离度最大的因果链。
6. Report Studio 能显示九章章节状态、引用字段、数据风险和导出状态。
7. 所有核心标题为问句或判断句。
8. 任何生成文字不出现 factPack 之外的数字。
9. HTML/PDF/PPTX 读取同一报告模型。

## 十、当前取舍

1. 先服务 To B 银行决策工作台，不新增独立 To C 行业筛选主入口。
2. 先做本地模板和因果链证据包，再恢复云端 DeepSeek 批量改写。
3. 先做 NIM、ROA、风险、价值创造四条高价值因果链，不追求一次覆盖所有银行经营问题。
4. 先用 ready 层和年报抓取聚合视图，不把 long format 明细直接塞进前台主画布。
5. 先保证 HTML/PDF/PPTX 同源和可用，再做 PPTX 高保真母版。
