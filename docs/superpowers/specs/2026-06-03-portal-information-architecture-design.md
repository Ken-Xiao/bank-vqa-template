# BenchmarkIQ Portal 信息架构与 Web Design 要求

日期：2026-06-03
阶段：Design Requirement v1
状态：Design Sample v1 已生成，待用户评审后进入 implementation plan

## 1. 设计命题

当前 Portal 的核心问题不是分析能力不足，而是能力外露过多。用户进入后同时看到选择银行、总览、专题、数据、报告、导出、治理、AI 文案、方法论和底稿线索，导致每一屏都像功能全集，而不是一条清晰的客户任务流。

本轮设计目标是把产品从“长页面功能堆叠”重构为“客户可导航的经营质量诊断工作台”：

> 8 分钟内完成：设定口径 -> 看到答案 -> 确认证据 -> 深钻专题 -> 导出上会材料。

## 2. 用户视角分层

### 2.1 行领导 / 董办主任

核心诉求：快速知道本行当前经营质量结论、主要压力来自哪里、会上应该讨论什么。

默认只需要看到：
- 一句总判断。
- 三个董事会问题。
- 四到六个关键证据。
- 一键进入正式报告。

不应默认看到：
- 字段完整性明细。
- 全部专题展开。
- 大段方法论。
- 表格型底稿。

### 2.2 战略 / 财务 / 风险部门

核心诉求：验证结论是否能站住，找到可行动的业务责任点。

默认入口应是：
- 证据地图。
- 专题深钻。
- 口径风险和指标来源。
- 对标银行差异解释。

### 2.3 分析师 / 后端数据同事

核心诉求：确认字段、口径、来源、缺失、三源校验和可替换规则。

默认入口应是：
- 数据复核页。
- 指标探索器。
- 三源字段血缘。
- 明细事实表和专题字段包。

## 3. 推荐信息架构

Portal 重构为 6 个一等页面，每个页面回答一个主问题。

| 页面 | 页面名称 | 主问题 | 默认用户 | 内容边界 |
|---|---|---|---|---|
| 0 | Launch / 设定口径 | 本次汇报分析谁、和谁比、按什么场景看？ | 全部用户 | 目标银行、银行类型、区域、对标组、年份、报告场景 |
| 1 | Executive Answer / 经营质量答案 | 目标银行现在最关键的经营质量判断是什么？ | 行领导、董办 | 总判断、关键 KPI、三大董事会问题、首要行动 |
| 2 | Evidence Map / 证据地图 | 这个判断由哪些证据支撑？ | 董办、部门负责人 | 同业位置、异动归因、PB/资本市场信号、证据置信度 |
| 3 | Topic Hub / 专题中心 | 哪些专题需要进一步深钻？ | 部门负责人、分析师 | 盈利、息差、风险、资本、PB、零售、ALM、IFRS9 等专题卡 |
| 4 | Topic Detail / 专题深钻 | 单个专题的原因、证据和行动是什么？ | 部门负责人、分析师 | 每页只放一个专题，左图右述，下方底稿 |
| 5 | Report Studio / 报告工作室 | 如何把当前结论变成可交付材料？ | 董办、分析师 | 报告预览、版本切换、章节开关、HTML/PDF/PPTX 导出 |
| 6 | Data & Validation / 数据复核 | 这些数字来自哪里，口径是否可信？ | 分析师、后端同事 | 三源校验、字段血缘、指标探索器、数据质量事件 |

说明：页面 6 是高级工作台，不进入行领导默认路径，但在所有证据组件上通过“口径/来源”入口可达。

## 3.1 用户确认后的产品决策

1. **默认首页直接进入 Launch / 设定口径**：不再先展示最近项目或功能总览；最近项目放在二级入口。
2. **首屏语言随 Launch 场景改变**：用户选择董事会版、资本市场版或管理层行动版后，经营质量答案页的主标题、解释侧重和行动建议同步切换。
3. **Data & Validation 作为左侧一级页面**：数据复核不是隐藏能力，保留为分析师和后端对接的一等入口，同时不干扰客户默认路径。
4. **Report Studio 保留报告结构编辑能力**：不只做章节开关和导出，还保留章节排序、版本切换、文案编辑和专题取舍能力。

## 4. 全局导航设计

### 4.1 顶部 Global Bar

高度维持 48px，保留现有深蓝专业风格。

必须常驻：
- 品牌：BenchmarkIQ。
- 当前上下文：目标银行、年份、对标组摘要。
- 诊断状态：口径已设定、证据已生成、报告待导出。
- 快捷动作：刷新诊断、导出、保存项目。

不在顶部堆叠：
- 全部导出按钮。
- 全部数据按钮。
- 方法论入口。
- 所有专题入口。

### 4.2 左侧 Page Rail

在桌面端采用左侧竖向导航，代替当前长页面中的多个横向 Tab 和跳转条。

建议导航：
1. 设定口径
2. 经营质量答案
3. 证据地图
4. 专题中心
5. 报告工作室
6. 数据复核

视觉要求：
- 使用图标 + 短标签。
- 当前页高亮用蓝色左边线或底色，不使用大面积渐变。
- 左侧导航宽度控制在 72-180px，可在紧凑模式折叠为图标。

### 4.3 页面内跳转

页面内不再做十几个锚点，而只保留：
- 页面主问题。
- 关键 CTA。
- 证据卡深链。
- 返回上一级。

每个深钻页需要有 breadcrumb：
`经营质量答案 / 证据地图 / 专题中心 / 具体专题`

## 5. 页面级设计要求

### 5.1 Launch / 设定口径

目标：一屏内完成分析边界设定，避免继续向下滚动才能选完。

布局：
- 左侧 40%：目标银行选择。
- 右侧 60%：对标组、年份、分析场景和推荐理由。
- 底部横向动作条：生成诊断、加载最近项目、保存口径。

选择银行流程：
1. 先选银行类型：大行、股份行、城商行、农商行。
2. 若选择城商行/农商行，再选区域。
3. 再显示目标银行列表和搜索。
4. 对标组根据类型和区域自动推荐，用户可展开调整。

设计要求：
- 不使用长表单。
- 目标银行列表最多展示 12 个，剩余通过搜索。
- 对标银行默认只展示推荐组摘要，展开后再编辑。
- 高级筛选默认折叠。

### 5.2 Executive Answer / 经营质量答案

目标：客户进入分析后第一眼看到“答案”，不是看到一堆模块。

首屏结构：
- 顶部一句 governing thought，使用判断句，不超过 42 个中文字符。
- 第二行 4 个关键 KPI：ROA、NIM、NPL、PB 或资本余量。
- 中部 3 个董事会问题，每个问题绑定一句系统判断。
- 右侧或底部首要行动建议，不超过 3 条。

设计要求：
- 首屏最多 6 张信息卡。
- 不显示方法论解释。
- 不显示大表。
- 每个卡片必须有“结论句 + 证据数字 + 查看证据”。

### 5.3 Evidence Map / 证据地图

目标：用一屏说明“为什么系统这么判断”。

建议分为三组证据：
1. 同业位置：目标银行在对标组和类型均值中的位置。
2. 异动归因：本期变化最大的 3-5 个指标，以及可能原因。
3. 资本市场信号：PB、股息率、换手率、市值、ROA、风险和资本余量联动。

设计要求：
- 异动雷达不再横向拉长，改为纵向证据流。
- 每条异动必须包含：指标、变化方向、幅度、可能原因、影响判断。
- 同业位置不展示过长排行榜，默认只展示目标银行、对标均值、类型均值、最好/最弱同业。
- PB 分析与异动归因纵向排列，避免并排拥挤。

### 5.4 Topic Hub / 专题中心

目标：把专题从“全展开内容”变成“问题目录”。

专题分组：
- 经营质量：盈利真实性、净利润归因、非息质量、现金流质量。
- 息差与负债：息差防守、存贷深钻、资产负债重定价、流动性期限错配。
- 风险与资本：风险确认、零售风险、IFRS9 三阶段、信用减值与核销回收、资本消耗。
- 市场与估值：PB 归因、资本市场信号、股息与交易活跃度。
- 数据复核：三源校验、字段覆盖、口径风险。

每个专题卡只展示：
- 一个问题式标题。
- 一句当前结论。
- 一个核心数字或小图。
- 数据置信度标签。
- 进入深钻按钮。

不在专题中心展示：
- 长正文。
- 大表格。
- 三版本 AI 文案。
- 完整事实包。

### 5.5 Topic Detail / 专题深钻

目标：每个专题页只讲一个故事。

统一页型：
- 顶部：专题问题 + 当前判断 + 数据置信度。
- 主体：左侧主图，右侧 Insight Panel。
- Insight Panel：Situation、Complication、Evidence、Recommendation。
- 底部：事实表、口径说明、引用字段。

设计要求：
- 每个专题只有一个主图，辅助图进入折叠区。
- 右侧 Insight Panel 固定宽度 360-420px。
- 表格默认折叠为“查看底稿”。
- 三版本语言稿放入“文案版本”抽屉，不默认占屏。

### 5.6 Report Studio / 报告工作室

目标：让导出从“按钮堆”变成“交付控制台”。

布局：
- 左侧 70%：报告页面预览。
- 右侧 30%：版本、章节、专题、导出格式控制。

报告版本：
- 董事会版：结论更短，强调风险与行动。
- 资本市场版：强调估值折价和市场信号。
- 管理层行动版：强调责任专题和节奏。

设计要求：
- PPTX 预览不再放在长页面底部。
- HTML/PDF/PPTX 共享同一报告内容树。
- 导出按钮最多 3 个：HTML、PDF、PPTX。
- 高级导出和数据导出进入二级菜单。

### 5.7 Data & Validation / 数据复核

目标：服务分析师和后端对接，不干扰客户默认路径。

布局：
- 左侧：指标/字段选择器。
- 中间：趋势、对标、分布和缺失情况。
- 右侧：来源、口径、三源差异、可替换策略。

设计要求：
- 三源数据不互相覆盖，显示为主数据、Tushare sidecar、年报抓取核验层。
- 字段必须标注单位、符号、合并/母公司、净额/总额、更新时间。
- 数据质量事件可以推送回证据地图，但详细过程留在数据页。

## 6. 视觉风格要求

本次设计延续现有 BenchmarkIQ 风格，不改成营销站，也不做大面积视觉装饰。

### 6.1 色彩

继续使用现有色板：
- Navy：`#061d3c`，用于标题、主数字、导航。
- Blue：`#0099d8`，用于主操作、链接、强调边线。
- Ink：`#1c2733`，用于正文。
- Muted：`#5d6670`，用于说明和次级信息。
- Good/Mid/Bad：`#1f9e6f`、`#d28f10`、`#c0392b`。

禁止：
- 大面积灰底导致正式报告失去颜色。
- 大面积渐变和装饰光斑。
- 单一蓝紫色系铺满。

### 6.2 密度

新增三种密度模式：
- Executive：只看答案和关键证据，默认模式。
- Analyst：展开专题、底稿、口径和指标探索。
- Draft：用于报告编辑和导出检查。

切换密度不改变数据，只改变默认展开层级。

### 6.3 卡片和表格

沿用现有版式规范：
- 内容卡圆角 6px。
- 按钮圆角 4px。
- 不做卡片套卡片。
- 表格使用 fixed layout 和 tabular nums。
- 中文不拆字，银行名列保留最小宽度。

## 7. 内容表达要求

每个模块必须满足“四件套”：
1. 结论句：直接说判断。
2. 证据数字：至少一个目标银行数字和一个参照数字。
3. 机制解释：说明为什么会这样。
4. 下一步入口：查看证据、进入专题、导出报告或复核口径。

禁止表达：
- “本模块用于分析……”
- “可以看出……”
- “需要进一步关注……”但不说明为什么。
- 只陈述方法，不绑定选定银行。

推荐表达：
- “目标银行的 PB 折价更像盈利质量折价，而不是单纯流动性折价。”
- “息差压力不是单点波动，主要来自资产端收益率下行快于负债端成本调整。”
- “零售风险的解释重点应从整体不良率转向具体产品贡献。”

## 8. 内容迁移规则

从默认长页面迁出：
- 方法论大段文字。
- 稳定基线检查。
- 项目治理流程。
- AI 产品模式说明。
- 全部专题完整正文。
- PPTX 预览。
- 数据字段完整性表。

迁入位置：
- 方法论进入 Data & Validation 或帮助抽屉。
- 项目治理进入 Report Studio 的项目菜单。
- AI 文案进入 Topic Detail 的文案抽屉。
- PPTX 预览进入 Report Studio。
- 字段完整性进入 Data & Validation。

保留在默认路径：
- 设定口径。
- 经营质量答案。
- 证据地图。
- 专题中心入口。
- 报告工作室入口。

## 9. Sprint 建议

### Sprint IA-0：信息架构确认

产出：
- 页面地图。
- 导航关系。
- 内容迁移清单。
- 每页验收标准。

验收：
- 用户能确认 6 个页面是否符合客户视角。
- 每个现有模块都有去向：保留、迁移、折叠或移除。

### Sprint IA-1：Portal Shell 与导航

范围：
- Global Bar 精简。
- 左侧 Page Rail。
- 页面路由状态。
- Breadcrumb 和深链。

验收：
- 用户不再依赖长滚动找内容。
- 任意页面能回到经营质量答案和报告工作室。

### Sprint IA-2：Launch + Executive Answer

范围：
- 选择银行页一屏重排。
- 经营质量答案首屏。
- 董事会三问题。

验收：
- 选择银行不被遮挡。
- 首屏不超过 6 张核心卡。
- 进入分析后第一眼是结论，不是方法。

### Sprint IA-3：Evidence Map + Topic Hub

范围：
- 同业位置压缩。
- 异动归因重排为纵向证据流。
- PB 分析与异动归因纵向排列。
- 专题中心卡片化。

验收：
- 异动不再长条拉伸。
- 每条异动有可能原因。
- 专题中心只显示摘要，不展开长内容。

### Sprint IA-4：Topic Detail + Report Studio

范围：
- 专题页统一 SCR 模板。
- 表格和文案抽屉化。
- 报告工作室重排。

验收：
- 每个专题页只有一个主问题和一个主图。
- 报告导出入口集中。
- PPTX 预览不再占用默认分析页空间。

### Sprint IA-5：Data & Validation

范围：
- 三源数据复核页。
- 字段血缘和口径标签。
- 数据质量事件与证据地图联动。

验收：
- 后端同事能按字段来源和口径对接。
- 客户默认路径不被字段表干扰。

## 10. 验收标准

产品体验验收：
- 用户进入后 10 秒内知道下一步点击哪里。
- 默认路径不需要滚动超过 3 屏即可进入报告工作室。
- 经营质量答案页首屏展示总判断、关键数字和董事会问题。
- 证据地图不出现超过 6 行的大表。
- 专题中心不显示长正文。
- Report Studio 集中承载所有正式交付控制。

视觉验收：
- 页面不出现大面积灰色正式报告。
- 无卡片套卡片。
- 无文字遮挡。
- 表格银行名不被拆成单字。
- 数字列对齐。
- 同一页面不超过 2-3 个字号层级。

内容验收：
- 每个模块都有结论句。
- 每个结论至少绑定一个目标银行数字和一个参照数字。
- 异动和偏离必须给出可能原因。
- PB、零售风险、息差、存贷深钻等专题可以进入报告和专题深钻。
- 数据口径不确定时，语言自动降级，且能进入数据复核页查看原因。

## 11. 待评审问题

1. 默认首页是否直接进入 Launch，还是保留一个最近项目入口？
2. 经营质量答案页默认面向“董事会版”，是否需要用户在 Launch 选择“资本市场版/管理层行动版”后改变首屏语言？
3. Data & Validation 是否作为左侧一级页面，还是只从证据卡进入？
4. Report Studio 是否需要保留报告结构编辑能力，还是先只做章节开关和导出？

## 12. 自检
