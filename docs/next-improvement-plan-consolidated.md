# BenchmarkIQ 下一步改进计划合并版

更新时间：2026-06-06

## 汇总口径

本计划合并了早期产品设计全案、V1/V2/V3/V4 PRD，近期 V5/V6 PRD，本轮新增的 BenchmarkIQ V3/V4、投行级价值诊断工作台、数据扩展与图表体系，以及最新 PRD v7.0/v7.1 的“Precision, not heft”产品重构要求。结论是：下一阶段不应继续零散加模块，而应围绕“8 分钟从数据到报告”的核心价值链做收敛。

说明：本轮已按 `superpowers` 插件提供的多线程开发管理方式复核计划：主线程负责排期收敛，PRD 评审子线程检查优先级和遗漏，工程评审子线程检查实现顺序、模块依赖和验收风险。本计划按“需求池 -> 工作流 -> 里程碑 -> 依赖 -> 验收 -> 暂缓项”组织。

## 总体优先级判断

当前产品已经具备：SPARC/VQA 总览、对标组选择、专题工作台、正式报告、V4/V5/V6 深钻和价值归因。真正限制产品进入试点和规模化交付的不是缺一个新图，而是以下五个断点：

1. 报告交付物还未彻底同源，HTML/PDF/PPTX 页序和视觉一致性仍需统一。
2. 产品入口仍像“功能全集”，而不是“设定口径 -> 看结论 -> 出报告”的三步主线。
3. 总览页和分析页信息密度仍偏高，董事会入口需要减负到“1+3+N”结构。
4. 数据页还不是分析师验证工作台，缺少“选指标 -> 看趋势/对标/口径”的闭环。
5. 口径风险、语言强度和 AI/模板叙事尚未完全联动。

因此下一步改造顺序建议为：

**先重构产品壳与导航 -> 再统一交付物 -> 再压缩内容主线 -> 再升级语言与深度分析。**

经过 superpowers 多线程复核后，对执行顺序做一处关键调整：**PPTX/导出不能整体放到后期**。应拆成两层：

1. **导出可用性 P0**：HTML 有正式版式、PDF 不白版且分页可读、PPTX 至少有页序/标题/图表/关键证据块。
2. **PPTX 高保真/矢量化 P2**：在报告内容树稳定后，再做完全矢量化、复杂版式和更高保真母版。

## 本轮新增 PRD 纳入后的排期校准

新增 PRD 把产品方向进一步明确为：从“银行位置比较工具”升级为“投行级价值诊断工作台”。新增需求不改变总排序，但会提高三类事项的优先级：

1. **机制归因提前**：DuPont、净利润归因瀑布、NIM归因、多基准线不再只是高级增强，而是报告能否像咨询交付物的核心。
2. **数据可信度提前**：257 字段利用率、口径风险、指标手册、异常检测、信用风险迁徙，需要先形成可验证的数据工作台。
3. **表达体系提前**：断言型标题、关键数字锚定、So What、七类页型、图表标注层，要进入报告和 PPTX 的基础规范。

## PRD v7/v7.1 增补后的产品原则

最新 PRD 的核心判断是：当前产品不是能力不足，而是能力外露过多。下一轮升级的主命题不是“加什么”，而是“怎么减、怎么藏、怎么排序”。对标 McKinsey、KPMG Peer Bank 和 360factors 后，产品目标被重新表述为：

**在一位银行董办主任收到年报数据后的 48 小时内，帮助其在 8 分钟内完成：选定口径 -> 看到诊断结论 -> 确认证据 -> 导出可上会材料。**

## PRD v8 增补：从可交付工具升级为全屏决策工作台

本轮新增 PRD 的核心判断是：当前产品的底层能力已经较强，下一步不是继续增加“重模块”，而是建立更强的启动体验、洞察推理、全屏呈现和数据互动能力。对标 McKinsey GBAR 的 “Precision, not heft”、KPMG Peer Bank 的 3-click insight、360Factors Peer Insights 的自动 peer benchmarking 后，产品下一阶段应从“能生成报告”升级为“能在董事会场景中直接形成判断、解释机制、驱动行动”。

### v8 产品原则

1. **60 秒启动**：用户不应先学习工具，而应先选银行、自动生成 peer、看到第一句判断。
2. **洞察优先**：每个专题必须从“当前值 + 变化方向 + 机制解释”三角生成结论，不再只复述指标。
3. **一套前台语言**：SPARC/VQA 继续作为计算层，但用户端统一呈现为“经营质量诊断”，避免三套五维框架并列。
4. **董事会对话语言**：标题和正文改成“问题-紧张-方向”，每页回答一个董事会真正会问的问题。
5. **全屏工作台**：在线体验从窄卡片看板升级为左导航、中主画布、右上下文面板的专业分析工作台。
6. **数据从静态对标走向动态推演**：What-if、同业轨迹、热力图和时间序列联动成为下一阶段的数据应用重点。

### v8 对当前排期的影响

v8 不推翻 Sprint 7/8 的成果，而是把下一阶段从“AI与规模化专项”进一步拆清楚：

1. **Sprint 9 优先做体验与语言的高感知改造**：60 秒启动、问题式标题、洞察三角、全屏宽度治理。
2. **Sprint 10 再做动态数据工作台**：时间序列叠加、对标矩阵热力图、What-if 全图表联动。
3. **Sprint 11 才进入高阶推理与规模化**：经济利润、诊断决策树、具名同业案例库、云端 LLM 推理增强。

### v8 关键取舍

- 先做 **60 秒启动 + 问题式标题 + 洞察三角**，因为这三项工程量相对可控，但用户感知最强。
- 全屏三栏工作台应作为版式治理继续推进，但必须保留现有 `setup / analysis / report / drawer` 状态机，不回到复杂 Tab 平铺。
- 经济利润、诊断决策树、同业轨迹聚类价值高，但需要更多数据解释和验证，放在 Sprint 11。
- 云端 LLM 不应抢先于事实包、引用审计和语言模板治理，否则容易生成漂亮但不可辩护的文字。

## PRD v9 增补：三源数据驱动的深钻专题体系

本轮新增数据盘点确认：当前产品已经拥有三类数据源，且各自适合承担不同角色。

1. **原有主数据**：342 条 bank-year，覆盖 2020-2025，是 Portal 当前的主数据底座，适合做历史趋势、对标均值、默认诊断和报告主线。
2. **Tushare sidecar**：360 条 bank-year，适合补标准三表、市场估值、股息率、总市值、换手率和部分衍生验证字段。
3. **年报抓取数据**：已生成去重明细库，覆盖 51 家标准化银行、51 个 2025 年 bank-year、20,730 条明细记录；原始目录当前有 728 个 CSV，其中包含杭州银行重复目录，适合补监管指标、附注明细、期限结构、零售分产品不良、IFRS9 三阶段和核销回收等专题深钻。

因此下一阶段的数据策略不应是“三源互相替换”，而应是：

**原有主数据做主表，Tushare 做市场和标准三表补充，年报抓取做 2025 年报核验层和附注明细层。**

### v9 专题页形态

所有新增专题统一采用 SCR + Evidence Drill 的页型：

1. **Situation**：本行当前位置、变化方向、同业位置。
2. **Complication**：差异来自规模、价格、结构、风险确认还是市场定价。
3. **Evidence**：核心图表 + 字段来源 + 样本 N + 口径风险。
4. **Recommendation**：0-3 月、3-6 月、6-12 月行动建议。
5. **Drill**：右侧可展开明细表，展示银行、年份、原始页码和抓取状态。

### v9 对排期的影响

v9 不替代 Sprint 9/10 的产品体验计划，而是把 Sprint 11 之后的数据深钻专项拆清楚：

1. **Sprint 11A：三源口径治理与年报核验层**，先解决数据是否可信、能不能进入报告。
2. **Sprint 11B：零售风险归因 + 资本市场信号增强**，优先做 Portal 已经有入口但数据没用深的专题。
3. **Sprint 12A：资产负债重定价 + 流动性期限错配**，扩展 NIM 防守和 ALM 专题。
4. **Sprint 12B：IFRS9 三阶段 + 信用减值/核销回收 + 投资结构波动**，扩展风险和利润质量专题。
5. **Sprint 13：后端数据仓库对接与明细事实表**，将抓取 long format 正式沉淀为可更新的数据层。

### v9 关键取舍

- **先做核验层，不先覆盖主表**：年报抓取与主数据存在单位、符号和口径差异，必须先出差异解释。
- **先做 Portal 已有入口的专题**：零售风险、PB/资本市场、NIM/ALM 已经有用户感知入口，新增数据接入后价值最直接。
- **明细表先做后端事实表，不急着塞进前端宽表**：IFRS9、期限桶、行业不良、核销回收天然是 long format，前端只读取聚合视图和可钻取明细。
- **Tushare 市场字段继续走 sidecar**：市场行情字段不应混入年报抓取口径，避免年末快照与财报披露口径混淆。

## PRD v11 增补：董事会故事线与交叉信号引擎

本轮新增两份材料进一步明确：BenchmarkIQ 不应只是“银行指标对标工具”，而应升级为“银行经营质量诊断工作台”。产品价值不在于展示更多指标，而在于帮助董事会和管理层减少决策不确定性：知道本行在哪、为什么在这里、下一步先改什么、改善后价值会如何变化。

详见独立 PRD：`docs/prd-board-storyline-cross-signal-engine.md`。

### v11 产品原则

1. **先服务 To B 银行决策工作台**：To C 投资分析能力暂不做独立主入口，保留资本市场语言和估值深钻。
2. **三幕九章故事线**：用 Situation / Complication / Resolution 串起总览、位置、异动、盈利、息差、风险、资本、行动和情景推演。
3. **交叉信号优先于单点指标**：每个判断都应回答“为什么”，而不是只说某指标高于或低于对标。

## v7 重新规划后的 4 个开发批次

### Batch A：能呼吸的产品壳

目标：让用户打开页面后 10 秒内知道先做什么、当前在哪、下一步去哪里。

范围：
- 48px Global Bar。
- `setup / analysis / report / drawer` 状态机。
- Step 1 设定口径页重构。
- Step 2 一屏一焦点骨架。
- Step 3 报告交付页骨架。
- 数据/复核/项目/AI 工具箱抽屉入口。
- Design Token 与布局约束。

验收标准：
- 进入页面默认只看到设定口径流程，不再被 6 个 Tab 和多层面板包围。
- 确认口径后进入 Step 2，Global Bar 显示银行、年份、VQA 分、信号和导出入口。
- 报告、数据、复核、项目管理都能通过 Step 或工具箱进入，但不抢默认视野。

### Batch B：精准的默认内容

目标：把默认分析页从“功能陈列”改成“1+3+N”的董事会故事线。

范围：
- 30 秒诊断合并：Client Brief + President Summary + Command Center。
- 三个董事会讨论问题。
- 同业位置卡：SPARC 五灯号 + 可展开雷达。
- Top Changes & Deviations。
- PB 估值答案。
- 专题入口折叠卡：SCR 四段论。
- 行动建议：0-3 / 3-6 / 6-12 月。

验收标准：
- Step 2 首屏可以在 30 秒内读完核心判断、VQA 信号、最弱维度和三个问题。
- 不滚动超过 3 屏即可看完同业位置、异动偏离、PB 估值答案。
- 专题深钻默认折叠，但每个专题有一句判断、SCR 逻辑和进入完整图表的入口。

### Batch C：语言、逻辑和证据穿透

目标：让每个页面都有判断、有证据、有机制解释、有管理含义。

范围：
- 判断句标题替换描述句标题。
- 本地语言强度 Level 1/2/3。
- z-score 与口径风险驱动语气分层。
- 章节桥梁句。
- 核心问题贯穿所有深钻页。
- 专题全屏深钻视图。
- 三版本 AI 解读：董事会版 / 资本市场版 / 管理层行动版。

验收标准：
- 所有核心标题都是结论句，不再只是模块名称。
- 严重偏离指标能进入执行摘要和讨论问题；普通指标留在专题或附录。
- 每个专题深钻页都能解释“为什么要看下一节”。

### Batch D：交付质量和规模化能力

目标：让 HTML/PDF/PPTX 真正共享同一个报告母版，并支持客户化编辑。

范围：
- Single-source report model。
- PDF 分页修复。
- PPTX 核心图表矢量化。
- HTML/PDF/PPTX 页序一致性。
- 报告结构编辑器。
- 版本切换：董事会版 / 资本市场版 / 管理层版。
- 对标组敏感性测试。
- 历史版本和导出留痕。

验收标准：
- PDF 不白版、不截断关键表格。
- PPTX 打开后至少核心图表清晰、标题和证据块可编辑。
- 用户能选择报告版本、开关专题并导出一致页序的交付物。

## 最重要的取舍

如果只能先做 3 件事：

1. **三步产品主线和状态机**：解决“功能很多但不知道先做什么”的最大短板。
2. **1+3+N 默认诊断页**：让董事会入口从“工具页”变成“决策页”。
3. **报告同源和导出质量**：让 HTML/PDF/PPTX 读取同一内容母版，避免交付不稳定。

如果只能先做 1 件事：

**先做“能呼吸的产品壳”。**
因为现在最大问题不是缺少能力，而是用户进入后被能力淹没。没有清晰状态机和 Step 主线，后续再做报告同源、语言和图表都会继续堆在长页面里。

## 当前已覆盖但需继续打磨的能力

- SPARC/VQA 映射：已配置化，但需要文档化和报告解释。
- 经济利润/DDM/PB归因：已实现基础，但需要纳入市值管理完整页序。
- 异动雷达：已实现 V6 版，但可继续接入跨期异常检测事件。
- 零售风险、存贷深钻、PB专题：已进入专题和正式报告，但图表页和 PPTX 同源仍需统一。
- 行长版摘要：已有基础，但应成为总览首屏的一部分，而不是散在模块中。

#### 当前最明确的剩余缺口

1. **浏览器级 QA 缺口**：自动契约测试较完整，但真实浏览器截图、PDF 导出、PPTX 打开后逐页检查仍未闭环。
2. **页面仍偏长**：导航、地图、控制台已经可以缩小，但首页和报告前置区还需要按“首屏答案优先”重新排版。
3. **语言体系未全量覆盖**：CEAM 结构已经有底座，仍要把每章标题、图表注释、So What、管理含义全部接入。
4. **对标组敏感性不足**：当前能解释为什么选，但还不能系统证明“换一两家银行结论是否仍成立”。
5. **PPTX 高保真未进入主线**：目前是可交付基础版，矢量化和复杂母版应放到 report model 稳定后做。

### 验收清单

每个 Sprint 完成前至少通过：

1. `node --check` 覆盖新增和修改的 JS 文件。
2. `analysis_rules.json` 等配置文件可正常解析。
3. `git diff --check` 无空白或冲突标记问题。
4. 本地静态服务可打开首页，选择目标银行后不报错。
5. 至少用一家城农商、一家股份行、一家大行跑通：总览、专题、正式报告、HTML/PDF/PPTX 导出。
6. 对关键页做目视验收：首屏不拥挤、报告有颜色、PDF不白版、PPTX有图表和图片、专题语言不是空泛方法论。
7. 固定三家回归样本：一家大行、一家股份行、一家城农商；每次发版都跑总览、专题、正式报告和三类导出。

### Superpowers 复核后的执行结论

1. **计划方向成立**：下一步应从“继续加模块”转向“少按钮、大屏幕、一脚油门到报告”的产品主线。
2. **工程顺序重排**：先做 `appMode` 状态机和 Global Bar，再迁移 Step 1/2/3；否则新需求会继续堆在旧长页面中。
3. **导出仍要前移，但不是第一步**：报告同源和 PDF/PPTX 修复放入 Sprint 7D，在产品壳和默认诊断页稳定后推进。
4. **AI 先本地增强，后云端接入**：先用 z-score、口径风险和 CEAM 做语言强度分层，云端 LLM/Kaia 放入 Sprint 8。
5. **PPTX 拆两层做**：P0/P1 做可交付版，P2 做矢量化和高保真母版。

## PRD v10 增补：Portal 信息架构与客户视角页面重排

本轮新增判断：当前 Portal 的主要瓶颈已经从“缺分析能力”转为“默认路径过长、页面承载过多、客户导航感不足”。下一步不应继续把 PB、异动、专题、数据复核和导出控制塞在同一条长页面里，而应重构为多页面工作台。

设计文档：`docs/superpowers/specs/2026-06-03-portal-information-architecture-design.md`

Web Design 样稿：`docs/design-mockups/portal-ia-v10-mockup.html`

### v10 产品原则

1. **每页只回答一个主问题**：设定口径、经营质量答案、证据地图、专题中心、专题深钻、报告工作室、数据复核各自独立。
2. **默认路径服务客户，深层路径服务分析师**：行领导默认看答案和证据，字段、口径、方法论和底稿进入数据复核或抽屉。
3. **从长滚动改为可导航工作台**：保留 48px Global Bar，新增左侧 Page Rail、breadcrumb 和证据深链。
4. **把能力收起来，而不是删掉**：方法论、AI 文案、PPTX 预览、三源校验和完整事实表迁出默认路径。
5. **设计继续沿用 BenchmarkIQ 风格**：深蓝、咨询感、紧凑卡片、固定表格列宽、数字对齐，不做营销式 hero 或大面积装饰。

### v10 目标页面

| 页面 | 主问题 | 内容范围 | 默认角色 |
|---|---|---|---|
| Launch / 设定口径 | 本次分析谁、和谁比、按什么场景看？ | 银行类型、区域、目标银行、对标组、年份、场景 | 全部用户 |
| Executive Answer / 经营质量答案 | 当前最关键的经营质量判断是什么？ | 一句总判断、关键 KPI、三大董事会问题、首要行动 | 行领导、董办 |
| Evidence Map / 证据地图 | 判断由哪些证据支撑？ | 同业位置、异动归因、PB/资本市场信号、证据置信度 | 董办、部门负责人 |
| Topic Hub / 专题中心 | 哪些专题需要深钻？ | 盈利、息差、风险、资本、PB、零售、ALM、IFRS9 等专题入口 | 部门负责人、分析师 |
| Topic Detail / 专题深钻 | 单个专题的原因、证据和行动是什么？ | 左图右述、SCR 解释、事实表、引用字段 | 分析师、部门负责人 |
| Report Studio / 报告工作室 | 如何形成可交付材料？ | 报告预览、版本切换、章节开关、HTML/PDF/PPTX 导出 | 董办、分析师 |
| Data & Validation / 数据复核 | 数字来自哪里，口径是否可信？ | 三源校验、字段血缘、指标探索器、数据质量事件 | 分析师、后端同事 |

### v10 Sprint 建议

| Sprint | 目标 | 主要改造 | 验收标准 |
|---|---|---|---|
| IA-0 | 信息架构确认 | 页面地图、导航关系、内容迁移清单、每页验收标准 | 每个现有模块都有明确去向：保留、迁移、折叠或移除。 |
| IA-1 | Portal Shell 与导航 | Global Bar 精简、左侧 Page Rail、页面路由状态、breadcrumb | 用户不再依赖长滚动找内容，任意页面能回到答案页和报告工作室。 |
| IA-2 | Launch + Executive Answer | 选择银行页一屏重排、经营质量答案首屏、董事会三问题 | 选择银行无遮挡；分析首屏不超过 6 张核心卡；第一眼是结论。 |
| IA-3 | Evidence Map + Topic Hub | 同业位置压缩、异动纵向证据流、PB 与异动纵排、专题卡片化 | 异动不再长条拉伸；每条异动有可能原因；专题中心不展开长内容。 |
| IA-4 | Topic Detail + Report Studio | 专题统一 SCR 模板、表格和文案抽屉化、报告工作室重排 | 每个专题只有一个主问题和主图；导出入口集中；PPTX 预览迁出默认页。 |
| IA-5 | Data & Validation | 三源复核页、字段血缘、口径标签、质量事件联动 | 后端同事能按字段来源和口径对接，客户默认路径不被字段表干扰。 |

### v10 对当前优先级的影响

- **下一轮优先做 IA-0/IA-1**，先确认页面地图和导航壳，再进入页面重排。
- **选择银行页继续作为 IA-2 的重点**，目标是银行类型、区域、目标银行、对标组和场景一屏完成。
- **异动雷达和同业位置纳入 IA-3**，从长条展示改为更短的证据流，并增加可能原因。
- **Report Studio 纳入 IA-4**，解决交付控制台挡住主体、PPTX 预览占位和导出按钮堆叠问题。
- **三源数据和 Tushare 口径继续保留在 Data & Validation**，不干扰客户默认浏览路径。

## 不建议下一轮优先做的事项

- 260 家发债银行覆盖：数据管道和口径维护成本高，先把 57 家上市银行做扎实。
- 完整资本消耗预测：需要分红率、RWA计划、利润留存等数据，先保留 What-if。
- 全量 PPTX 矢量化：价值高但工程风险大，等报告同源后再做。
- 大规模预警通知：没有稳定数据更新管道前，容易做成空壳。
- 复杂固定效应回归和同业轨迹聚类：价值高，但需要先把时间序列叠加、热力图和 What-if 联动做成稳定交互，再进入建模专项。

## 一句话结论

下一阶段的产品目标应从“继续加分析模块”转为“把能力收进一条可理解、可操作、可交付的主线”。最优路线是：

**产品壳与状态机 -> 1+3+N 默认诊断 -> SCR专题与语言强度 -> 报告同源与导出质量 -> AI与规模化专项。**
