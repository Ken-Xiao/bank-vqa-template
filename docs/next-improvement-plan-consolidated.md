# BenchmarkIQ 下一步改进计划合并版

更新时间：2026-05-31

## 汇总口径

本计划合并了早期产品设计全案、V1/V2/V3/V4 PRD，近期 V5/V6 PRD，以及本轮新增的 BenchmarkIQ V3/V4、投行级价值诊断工作台、数据扩展与图表体系等 PRD。结论是：下一阶段不应继续零散加模块，而应围绕“可上会交付”和“客户自助完成分析”两条主线做收敛。

说明：本轮已按 `superpowers` 插件提供的多线程开发管理方式复核计划：主线程负责排期收敛，PRD 评审子线程检查优先级和遗漏，工程评审子线程检查实现顺序、模块依赖和验收风险。本计划按“需求池 -> 工作流 -> 里程碑 -> 依赖 -> 验收 -> 暂缓项”组织。

## 总体优先级判断

当前产品已经具备：SPARC/VQA 总览、对标组选择、专题工作台、正式报告、V4/V5/V6 深钻和价值归因。真正限制产品进入试点和规模化交付的不是缺一个新图，而是以下四个断点：

1. 报告交付物还未彻底同源，HTML/PDF/PPTX 页序和视觉一致性仍需统一。
2. 总览页和分析页信息密度仍偏高，董事会入口需要减负。
3. 数据页还不是分析师验证工作台，缺少“选指标 -> 看趋势/对标/口径”的闭环。
4. 口径风险、语言强度和 AI/模板叙事尚未完全联动。

因此下一步改造顺序建议为：

**先统一交付物 -> 再重构首页决策漏斗 -> 再补数据/口径工作台 -> 再升级语言与深度分析。**

经过 superpowers 多线程复核后，对执行顺序做一处关键调整：**PPTX/导出不能整体放到后期**。应拆成两层：

1. **导出可用性 P0**：HTML 有正式版式、PDF 不白版且分页可读、PPTX 至少有页序/标题/图表/关键证据块。
2. **PPTX 高保真/矢量化 P2**：在报告内容树稳定后，再做完全矢量化、复杂版式和更高保真母版。

## 本轮新增 PRD 纳入后的排期校准

新增 PRD 把产品方向进一步明确为：从“银行位置比较工具”升级为“投行级价值诊断工作台”。新增需求不改变总排序，但会提高三类事项的优先级：

1. **机制归因提前**：DuPont、净利润归因瀑布、NIM归因、多基准线不再只是高级增强，而是报告能否像咨询交付物的核心。
2. **数据可信度提前**：257 字段利用率、口径风险、指标手册、异常检测、信用风险迁徙，需要先形成可验证的数据工作台。
3. **表达体系提前**：断言型标题、关键数字锚定、So What、七类页型、图表标注层，要进入报告和 PPTX 的基础规范。

## 合并后的 4 个开发批次

### Batch A：交付物同源与董事会入口

目标：让输出材料先“稳”和“像正式报告”。

范围：
- Single-source formal report engine
- 统一页序和目录
- Print CSS 与 PDF 分页完善
- 总览页折叠机制
- 30 秒结论 + 三大董事会议题 + SPARC交通灯首屏

验收标准：
- HTML/PDF/PPTX 章节顺序一致。
- 总览首屏不滚动即可看到核心答案。
- 正式报告前 3 页能回答“这次董事会该讨论什么”。

### Batch B：口径可信度与数据工作台

目标：让客户相信数据，分析师能验证结论。

范围：
- 口径风险 L1-L4 与叙事/图表联动
- 数据Tab拆成“覆盖质量 / 口径可比性 / 指标探索器”
- Metric Explorer 单指标视图
- 指标口径手册 V1
- 跨期异常检测和数据质量事件

验收标准：
- L3/L4 指标不会进入强结论。
- 任一指标可在 1 分钟内查看目标银行、对标组、趋势、分位、口径说明。
- 报告附录能解释核心指标口径和数据边界。

### Batch C：专业级分析深度

目标：从“位置比较”升级到“机制归因”。

范围：
- DuPont 三级分解
- 净利润 YoY 归因瀑布
- 多基准线体系
- 资产质量诚实度矩阵代理版
- 对标组敏感性测试
- Gap Attribution Narrative

验收标准：
- ROE、ROA、NIM、净利润变化都能回答“差距来自哪个因子”。
- 报告每个专题至少包含一个机制解释，而不是只说高于/低于对标。
- 换对标组时能说明结论稳不稳。

### Batch D：报告编辑、PPTX和规模化能力

目标：让客户可以独立出材料，顾问可以规模化交付。

范围：
- 报告结构编辑器
- PPTX 图表高保真/矢量化专项
- 精细化 Prompt 模板库
- 历史报告版本对比
- 预警通知和数据更新监测
- 智能推荐对标组

验收标准：
- 用户能开关和排序报告章节。
- PPTX 打开后版式稳定、图表清晰。
- 同一项目多轮报告可留痕和对比。

## 最重要的取舍

如果只能先做 3 件事：

1. **报告同源和页序统一**：解决交付物不稳定这个最大短板。
2. **总览页减负和折叠机制**：让董事会入口从“工具页”变成“决策页”。
3. **Metric Explorer + 口径联动**：让分析师能验证、董办能解释、报告能自洽。

如果只能先做 1 件事：

**先做报告同源和统一页序。**  
因为所有高级分析最终都要进入报告；如果交付物不稳定，越加功能越容易变成更多不一致。

## 当前已覆盖但需继续打磨的能力

- SPARC/VQA 映射：已配置化，但需要文档化和报告解释。
- 经济利润/DDM/PB归因：已实现基础，但需要纳入市值管理完整页序。
- 异动雷达：已实现 V6 版，但可继续接入跨期异常检测事件。
- 零售风险、存贷深钻、PB专题：已进入专题和正式报告，但图表页和 PPTX 同源仍需统一。
- 行长版摘要：已有基础，但应成为总览首屏的一部分，而不是散在模块中。

#### 当前最明确的剩余缺口

1. **浏览器级 QA 缺口**：自动契约测试较完整，但真实浏览器截图、PDF 导出、PPTX 打开后逐页检查仍未闭环。
2. **页面仍偏长**：导航、地图、控制台已经可以缩小，但首页和报告前置区还需要按“首屏答案优先”重新排版。
3. **语言体系未全量覆盖**：CEAM 结构已经有底座，仍要把每章标题、图表注释、So What、管理含义全部接入。
4. **对标组敏感性不足**：当前能解释为什么选，但还不能系统证明“换一两家银行结论是否仍成立”。
5. **PPTX 高保真未进入主线**：目前是可交付基础版，矢量化和复杂母版应放到 report model 稳定后做。

### 验收清单

每个 Sprint 完成前至少通过：

1. `node --check` 覆盖新增和修改的 JS 文件。
2. `analysis_rules.json` 等配置文件可正常解析。
3. `git diff --check` 无空白或冲突标记问题。
4. 本地静态服务可打开首页，选择目标银行后不报错。
5. 至少用一家城农商、一家股份行、一家大行跑通：总览、专题、正式报告、HTML/PDF/PPTX 导出。
6. 对关键页做目视验收：首屏不拥挤、报告有颜色、PDF不白版、PPTX有图表和图片、专题语言不是空泛方法论。
7. 固定三家回归样本：一家大行、一家股份行、一家城农商；每次发版都跑总览、专题、正式报告和三类导出。

### Superpowers 复核后的执行结论

1. **计划方向成立**：下一步应从“继续加模块”转向“稳定交付系统 + 机制归因 + 咨询化表达”。
2. **工程顺序要前移导出**：先把 `formalReport` 或新的 `reportModel` 固化为唯一内容 contract，再让 HTML/PDF/PPTX 读取它。
3. **银行选择页很重要，但不应压过导出同源**：入口减负放在 Sprint 2，更符合工程依赖。
4. **字段热力图要前置**：零售风险、行业风险、信用迁徙等专题先判断“可直接做/可代理/暂缓”，避免排期建立在缺失字段上。
5. **PPTX 拆两层做**：P0/P1 做可交付版，P2 做矢量化和高保真母版。

## 不建议下一轮优先做的事项

- 260 家发债银行覆盖：数据管道和口径维护成本高，先把 57 家上市银行做扎实。
- 完整资本消耗预测：需要分红率、RWA计划、利润留存等数据，先保留 What-if。
- 全量 PPTX 矢量化：价值高但工程风险大，等报告同源后再做。
- 大规模预警通知：没有稳定数据更新管道前，容易做成空壳。

## 一句话结论

下一阶段的产品目标应从“继续加分析模块”转为“把分析模块组织成稳定、可信、可上会的交付系统”。最优路线是：

**交付同源 -> 首页减负 -> 数据/口径工作台 -> 机制归因 -> 报告编辑与PPTX专项。**
